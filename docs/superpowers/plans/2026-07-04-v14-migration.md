# Foundry V14 Migration — Technical Plan

> **For agentic workers:** Steps use checkbox (`- [ ]`) syntax for tracking. A
> live Foundry **V14** instance is available — every "verify" step is a real
> runtime check, not a guess. Do the verification steps *before* committing the
> code change they gate, so we never ship a speculative fix. Land work on
> `V14-migration`; push convention is `gh auth switch --user YeNov` before push,
> switch back to `yehornovakov` after (see memory `push-as-yenov-account`).

**Goal:** `starwarsffg` runs clean on Foundry VTT V14 (stable 14.359) with
`compatibility.verified: 14` declared, no `ReferenceError` from removed globals,
no legacy-Document-API throws, and no new console deprecation warnings that we
introduced.

**Branch:** `V14-migration` (off `main` at `a95a2c3f`). Currently one commit
ahead — `342736e9` "Namespace bare global helper calls under foundry.utils for
V14" — unpushed.

**Starting state (audited 2026-07-04).** The V2-full migration is complete; the
system is already ~95% V13-namespaced. Confirmed via code audit:

| Area | State |
|---|---|
| Bare util globals — object helpers (`mergeObject`, `setProperty`, `duplicate`, …) | ✅ Fixed in `342736e9` |
| Bare util globals — `randomID()` | ❌ **`342736e9` missed this helper.** 5 bare `randomID()` calls remain and throw `ReferenceError` on V14 — see Stage 1.2 |
| ApplicationV2 / DialogV2 / HandlebarsApplicationMixin | ✅ Used throughout; no `extends Application`, no `new Dialog` |
| `FFGFormApplication` | ✅ Custom class on `HandlebarsApplicationMixin(ApplicationV2)` — not a V1 remnant |
| Dice terms | ✅ All extend `foundry.dice.terms.DiceTerm`; `RollFFG extends Roll` |
| `renderTemplate`/`loadTemplates` | ✅ Namespaced under `foundry.applications.handlebars.*` |
| `TextEditor`/`FilePicker`/`ContextMenu`/`DragDrop`/`Tabs` | ✅ Namespaced under `foundry.applications.ux.*` / `.apps.*` |
| MeasuredTemplate / TinyMCE | ✅ None (TinyMCE only appears in harmless CSS-guard selectors) |

So this migration is a short cleanup, not a rewrite. The plan is ordered by
urgency: Stage 1 = things that block or throw on V14; Stage 2 = polish.

## Stage 0 — Baseline capture on V14 (do first)

Establishes the "before" picture so we can attribute every warning/error.

- [ ] **0.1 — you must raise `maximum` before you can launch at all.**
      `system.json` sets `"maximum": 13` **explicitly**, and `14 > 13`, so V14
      treats the system as **hard-incompatible** (not merely "unverified"): a
      world requires its system, so Foundry blocks the world from launching on
      the setup screen. (`verified` exceeded = soft warning + launch allowed;
      `maximum` exceeded = incompatible + blocked. Different fields, different
      severity.) The manifest number has **no effect on runtime** — it's purely
      the launch gate — so there is no value in "testing against the old
      declaration". Do Stage 1.1's `maximum` bump **first** so you can get in.
      (Behavior is stable through V11–V13; V14 is past the author's cutoff —
      confirm on the instance, but expect a block.)
- [ ] **0.1a** At load, watch specifically for a `ReferenceError` thrown from
      [groupmanager-ffg.js:7](../../../modules/groupmanager-ffg.js) — it is
      module-level (`const CanvasLayerClass = foundry?.canvas?.layers?.CanvasLayer
      || CanvasLayer;`) and imported eagerly by
      [swffg-main.js:28](../../../modules/swffg-main.js), so if
      `foundry.canvas.layers.CanvasLayer` moved/was removed in V14 the bare
      `CanvasLayer` fallback throws **before any smoke path runs** and kills init.
      This gates Stage 1.6.
- [ ] **0.2** Open the browser console. With the system loaded (compatibility
      mode if needed), exercise the smoke path and capture **all** errors +
      deprecation warnings to a scratch file:
  - Open each actor sheet type: character, minion, rival/nemesis (adversary),
    vehicle, homestead, plus a Codex sheet if the `codex-UI` flag is on.
  - Open each item sheet type (weapon, armour, gear, talent, specialization,
    force power, signature ability, item-modifier, item-attachment).
  - Open the item editor (`{{editor}}` inline ProseMirror mount) — **this is the
    `ProseMirror` global check** (Stage 1.4).
  - Run the OggDude importer (data-importer → import-helpers path).
  - Open the Group Manager (Destiny Tracker menu → Group Manager) and Grant XP.
  - Roll a skill and a weapon with sound flags enabled (`roll-builder.js`).
  - Drag a skill and a weapon to the hotbar, then **execute** both macros —
    **this is the generated-macro `.data.data` / `getData()` check** (Stage 1.3).
- [ ] **0.3** Save the captured console log to
      `docs/superpowers/plans/artifacts/2026-07-04-v14-baseline-console.txt`
      (create the dir). This is the checklist Stage 1 must drive to zero.

### 1.1 Declare V14 compatibility — **required FIRST, to launch at all**

Corrects an earlier version of this plan that deferred the bump to "last": with
`maximum` explicitly capped at 13, V14 blocks world launch entirely (see 0.1),
and the manifest has no runtime effect, so there's nothing to gain by delaying.

- [x] **done** — **Up front — raise the cap so the world launches.** Edit

### 1.3 Legacy `.data.data` Document access (latent throws)

Pre-existing bugs (broken since V11), not V14-specific, but they're legacy-API
throws surfaced by this audit. Fix as part of the V14 cleanliness pass.

- [x] **done — commit `325cf0e4`** — [embeddeditem-helpers.js:227](../../../modules/helpers/embeddeditem-helpers.js) —
      `ownedItem.data.data[modifierType]` reads `.data` on a real Item Document
      (`undefined` since V11 → throws). Lines 231-232 immediately below already
      use `ownedItem.system[modifierType]` correctly — mirror that:
      `ownedItem.system[modifierType].findIndex(...)`.
      Verify the branch is reachable (`modifierId` non-numeric) via 0.2's
      item-modifier interactions; if 0.2 never hits it, still fix (dead-simple,
      correct-either-way) but note reachability.
- [x] **done — commit `325cf0e4`** — **Generated skill macros — there are TWO branches, fixed both.** The
      hotbar drop handler emits skill-roll macro command strings in two places:
  - [macros.js:36-40](../../../modules/helpers/macros.js) (`entity.type === "skill"`)
    — uses **stale `.data.data`**: `ffgactor.data.data.skills[...]` /
    `ffgactor.data.data.characteristics[...]`. Rewrite to `ffgactor.system.skills`
    / `ffgactor.system.characteristics`.
  - [macros.js:67-80](../../../modules/helpers/macros.js) (`data?.data?.type === "skill"`)
    — already uses `.system.*` (no `.data.data` bug), but shares the next defect.
  - **Both** branches call `const actorSheet = ffgactor.sheet.getData();`
    **without `await`**. `getData()` is *intentionally bridged* on the FFG V2
    sheets ([ffg-document-sheet.js:163,188](../../../modules/apps/ffg-document-sheet.js)
    → `_prepareContext` calls `getData`) and
    [`ActorSheetFFG.getData` is **async**](../../../modules/actors/actor-sheet-ffg.js) —
    so un-awaited it hands `rollSkillDirect` a **Promise**, not sheet data.
    **Do not** swap it for `_prepareContext()` — `getData()` is the correct bridge
    and produces exactly the legacy sheet-data shape `RollBuilderFFG` /
    `rollSkillDirect` consume (same object the in-sheet roll handlers pass, e.g.
    `await this.getData({})` at actor-sheet-ffg.js:459). The fix is simply to
    `await` it: `const actorSheet = await ffgactor.sheet.getData();` in both
    generated command strings. (Foundry runs script-macro bodies as async, so
    top-level `await` is valid.)
- [ ] Re-run 0.2's dragged-macro execution for a skill dropped via **both** code
      paths (owned skill item drag *and* a `Transfer`-payload skill drag) to
      confirm each macro now rolls.

### 1.7 ChatMessage style enum — removed `CONST` **and** wrong field — **DONE**

Not caught by the static audit (it looked for globals/namespacing, not removed
`CONST` enums or field-schema changes). Surfaced on first V14 world launch from
the migration runner's `sendChanges()`. Took **two** fixes:

- [x] **commit `553a6809`** — `CONST.CHAT_MESSAGE_TYPES` (renamed to
      `CHAT_MESSAGE_STYLES` in V12; alias removed in V14, so `undefined.OTHER`
      threw) → `CHAT_MESSAGE_STYLES` at
      [swffg-migration.js:60,74](../../../modules/swffg-migration.js) and
      [actor-sheet-ffg.js:1810](../../../modules/actors/actor-sheet-ffg.js).
- [x] **commit `70cb5606`** — the deeper issue: the value was assigned to the
      **`type`** field, but in V14 ChatMessage `type` is the *document subtype*
      (a validated string), so `type: 0` threw `"0" is not a valid type for the
      ChatMessage Document class`. The style enum belongs in the **`style`**
      field. Changed `type:` → `style:` at all 4 sites (1780 was latent-broken,
      just never exercised). No code reads message `.type`/`.style` as a style,
      so write-side only; `OTHER` is the default style so behavior is unchanged.
- [x] Swept all `CONST.*` usage: no other removed enums. Still valid on V14:
      `ACTIVE_EFFECT_MODES`, `TOKEN_DISPOSITIONS`, `DOCUMENT_OWNERSHIP_LEVELS`,
      `USER_ROLES`, `JOURNAL_ENTRY_PAGE_FORMATS`, `TOKEN_TURN_MARKER_MODES`,
      `DEFAULT_TOKEN`.

### 1.10 Stack overflow walking item data — recursion into live Documents — **DONE**

Static-audit miss (same V14 root as 1.9: live Documents where plain data was
assumed). Clicking a specialization threw `Maximum call stack size exceeded` in
`findKeysIncludingStringRecursively`.

- [x] **done — commit `3f703a96`** — the item-sheet getData context now holds the
      live item Document (`data.item = data.document` at
      [item-sheet-ffg.js:153](../../../modules/items/item-sheet-ffg.js)); on V14
      `for...in` descends into its `EmbeddedCollection`s (see
      `EmbeddedCollection.hasOwnProperty` in the trace), whose documents
      back-reference their parent → infinite cycle. V13 never reached those
      internals, so it didn't overflow. Added an `_isWalkable()` guard to both
      recursive walkers ([embeddeditem-helpers.js:319/306](../../../modules/helpers/embeddeditem-helpers.js))
      so they descend only into plain objects/arrays, skipping Documents /
      DataModels / EmbeddedCollections. The `-=attr` keys they seek only ever
      live in plain data, so behavior is preserved.
