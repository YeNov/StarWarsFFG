> **For agentic workers:** Steps use checkbox (`- [ ]`) syntax for tracking. A
> live Foundry **V14** instance is available — every "verify" step is a real
> runtime check, not a guess. Do the verification steps *before* committing the
> code change they gate, so we never ship a speculative fix. Land work on
> `V14-migration`; push convention is `gh auth switch --user YeNov` before push,
> switch back to `yehornovakov` after (see memory `push-as-yenov-account`).

**Goal:** `starwarsffg` runs clean on Foundry VTT V14 (stable 14.359) with
`compatibility.verified: 14` declared, no `ReferenceError` from removed globals,
no legacy-Document-API throws, and no new console deprecation warnings that we
introduced.

**Branch:** `V14-migration` (off `main` at `a95a2c3f`). Currently one commit
ahead — `342736e9` "Namespace bare global helper calls under foundry.utils for
V14" — unpushed.

**Starting state (audited 2026-07-04).** The V2-full migration is complete; the
system is already ~95% V13-namespaced. Confirmed via code audit:

| Area | State |
|---|---|
| Bare util globals — object helpers (`mergeObject`, `setProperty`, `duplicate`, …) | ✅ Fixed in `342736e9` |
| Bare util globals — `randomID()` | ❌ **`342736e9` missed this helper.** 5 bare `randomID()` calls remain and throw `ReferenceError` on V14 — see Stage 1.2 |

So this migration is a short cleanup, not a rewrite. The plan is ordered by

### 1.13 Drag item → actor: `effect.duration.value must be an integer` — **DONE (verify)**

**Root cause = a V14 core-schema change.** V14 replaced the `ActiveEffect.duration`
model (`{seconds, rounds, turns, combat, …}`) with `{value, units, expiry,
expired}`, where `value` must be an **integer when `units` is set**. Diagnostic
dump of a live effect confirmed it:
`Weapon :: (inherent) :: {"value":null,"units":"seconds","expiry":null,"expired":false}`
— the invalid "units-without-value" state. V14 migrates legacy effects into this
shape on load; it's tolerated in place but **rejected on create**, so
`createEmbeddedDocuments("Item", …)` throws. Systemic, not one bad item: the
system puts an `(inherent)` effect on every modifier-bearing item.

**It's not just legacy data:** a *freshly created* V14 item already carries the
bad shape — it's V14's own default, tolerated as a default but rejected when the
same value is supplied *explicitly* (i.e. any time an effect is copied via
`toObject()` and re-created). So the drop path was only 1 of ~20 create sites
(`Actor.create` in importer/character-creator, gm-bridge grants, purchase/
transfer, `_onCreateAEs`, item-editor, …).

- [x] **commit `ae8f7e38`** — [ffg-actor-sheet.js:226](../../../modules/apps/ffg-actor-sheet.js)
      `_onDropItemCreate` strips any effect `duration` whose `value` isn't an
      integer (kept as belt-and-suspenders for the drop path).
- [x] **commit `3a889700` — central fix.**
      [active-effect-ffg.js](../../../modules/active-effects/active-effect-ffg.js)
      overrides `ActiveEffectFFG.migrateData` to delete a malformed core
      `duration` (value not an integer). `migrateData` runs **before validation
      on every construction path**, so Foundry applies a valid default
      everywhere — one fix covers all create/copy sites, and re-created effects
      persist the cleaned shape. Real integer durations are untouched; the system
      never uses core duration (it tracks its own in `system.duration`).
- [ ] **Verify on V14:** re-drag the weapon and other modifier items; also try
      an OggDude import / character build if handy — all should create clean.

### 1.14 `applyActiveEffects` override dropped the V14 phase argument — **DONE, verified**

Another static-audit miss (a V14 method-signature change, only visible at runtime).
V14 made `Actor#applyActiveEffects` **phase-based**: core now calls it with a
string phase identifier (`"initial"`, …) and `super.applyActiveEffects()` logs a
deprecation ("must be called with a string phase identifier … removed in V16")
when that argument is missing. Surfaced on **every actor prepare** (seen while
toggling a token status, but not specific to statuses).

- [x] **[actor-ffg.js:777](../../../modules/actors/actor-ffg.js)** — the override
      was declared `applyActiveEffects()` and called `super.applyActiveEffects()`,
      silently dropping the phase core passed. Changed to `applyActiveEffects(...args)`

      mutations (quantity scaling, force-pool math) are already idempotent — they
      derive from `_source`/live system values each call — so V14 invoking this
      once per phase is safe. Cross-compatible: V13 passes no args, V14 passes the
      phase. **Verified on V14:** the deprecation warning is gone after reload.
